# ADR-002：单租户优先、多租户兼容预留

**日期：** 2026-07-20  
**状态：** Accepted  
**决策者：** TrioBase 架构组

## 背景

TrioBase 已具备租户主数据管理、网关租户状态校验、`X-Tenant-Id` 上下文透传，以及部分业务表的 `tenant_id` 字段。然而，租户初始化、全服务自动数据过滤、数据库行级安全、套餐配额和跨租户隔离测试尚未完整落地。

当前阶段的首要目标是继续构建业务、工作流、数据和 AI 核心能力。此时把租户管理作为正式的多租户能力开放，会造成数据隔离能力与产品承诺不一致，并增加各模块的同步建设成本。

## 决策

当前阶段采用“单租户运行、多租户兼容预留”模式：

- 系统仅支持默认租户 `default` 作为正式运行租户。
- 所有用户、组织、权限、低代码、工作流、数据和 AI 能力均归属默认租户。
- 租户管理功能暂不作为正式产品能力对外开放。
- 保留现有租户服务、数据库 `tenant_id` 字段和租户上下文契约，不删除多租户兼容结构。
- 在正式完成多租户隔离验收前，不宣称系统支持生产级多租户。

## 当前阶段约束

### 请求与身份

- 登录凭证和网关下发的租户上下文统一为 `default`。
- 继续由网关校验默认租户状态，并向下游透传 `X-Tenant-Id`。
- 不信任客户端自行提交或覆盖的租户标识。
- 非明确的平台内部接口不得切换目标租户。

### 数据与业务

- 新增租户业务表应保留非空 `tenant_id` 字段，当前统一写入 `default`。
- Redis Key、Kafka 事件、Temporal 上下文、向量数据和分析数据应保留租户维度，当前值固定为 `default`。
- 新功能不得依赖创建额外租户才能运行。
- `plan_code`、`max_users` 和租户级设置暂不作为强制的功能开关或配额执行依据。

### 产品与界面

- 租户创建、停用和配置入口暂不面向普通用户开放；必要时仅供开发或平台维护使用。
- 当前 `ADMIN` 承担系统管理职责，暂不扩展平台管理员与租户管理员的完整分层模型。

## 不在当前范围

- 自动租户开通与初始化流程。
- 全微服务统一租户 SQL 拦截器。
- PostgreSQL Row Level Security。
- Schema、数据库或基础设施级物理隔离。
- 租户套餐、配额、计费和功能授权联动。
- 跨租户运维、租户迁移与租户级备份恢复。

## 未来启用多租户的准入条件

只有同时满足以下条件，才能把多租户标记为正式可用：

1. 所有租户业务表和唯一约束完成租户维度审计。
2. 所有数据访问路径具备统一、默认拒绝的租户过滤机制。
3. HTTP、Kafka、Temporal、定时任务、RAG 和分析链路均能可靠传播租户上下文。
4. 完成租户初始化、管理员创建、权限模板、组织根节点和基础配置的自动化流程。
5. 套餐、配额和租户设置存在明确的消费与执行组件。
6. 建立跨租户读取、修改、缓存、搜索和异步处理的隔离集成测试。
7. 完成安全评审、数据迁移方案和生产环境隔离验收。

## 后果

### 正面影响

- 团队可以集中建设 TrioBase 核心业务与 AI 能力。
- 当前产品能力与实际安全边界保持一致。
- 保留未来演进多租户所需的数据模型和上下文契约，降低后续改造成本。

### 负面影响

- 当前不能向多个互不信任的企业客户提供共享部署。
- 各模块仍需遵守租户字段和上下文预留约束，会产生少量额外开发成本。
- 后续正式启用多租户时，仍需进行一次全系统数据访问审计和专项改造。

## 复审触发条件

出现以下任一情况时，应复审本 ADR 并提出正式的多租户建设变更：

- 需要在同一部署中服务两个或以上互不信任的企业客户。
- 商业模式要求租户级套餐、配额或独立计费。
- 客户提出租户级数据驻留、备份、审计或合规要求。
- 系统准备进入面向外部客户的生产 SaaS 阶段。
