## Context

web-antd 页面直接组合 Ant Design Vue `Button`、`Space` 和容器样式。查询区、表格工具栏、行操作以及弹窗 footer 已形成若干相似模式，但没有统一组件或可测试契约，导致按钮顺序、间距、对齐和溢出策略不一致。

## Goals / Non-Goals

**Goals:**

- 用轻量布局组件统一按钮区域结构，不接管业务事件和权限判断。
- 固化查询区“重置 → 查询”、表单区“取消 → 主操作”和工具栏左右分区。
- 支持现有按钮通过 slot 渐进迁移，避免改写业务逻辑。
- 为布局组件和主要迁移页面提供自动化验证。

**Non-Goals:**

- 不替换 Ant Design Vue Button。
- 不变更后端权限、业务 Action、接口或状态机。
- 本次不一次性迁移演示页、认证页和高度定制的画布编辑器。
- 不自动推断按钮的业务语义或权限。

## Decisions

### 使用 slot 驱动的语义布局组件

新增 `shared/page-actions`，提供 `PageQueryActions`、`PageToolbar`、`TableActionBar` 和 `FormFooterActions`。组件只负责 DOM 结构、对齐、间距和响应式换行，调用方继续声明 Button、权限条件和事件。

选择 slot 而不是按钮配置数组，是为了保留现有 Vue 模板的类型、指令、Popconfirm 和业务组件组合能力，降低迁移风险。

### 顺序由组件槽位结构表达

`PageQueryActions` 暴露 `reset`、`submit` 槽位；`FormFooterActions` 暴露 `cancel`、`primary` 槽位；`PageToolbar` 暴露 `start`、`end` 槽位。具名槽位使规范可读且可测试，同时避免通过 CSS `order` 隐式重排造成键盘焦点顺序错误。

### 行操作采用渐进式规范

`TableActionBar` 统一横向间距、换行和对齐。本次迁移保持已有业务操作可见性，不强制把所有超过三个的操作立即改成下拉菜单，避免改变操作可发现性；新增页面应遵循“查看、编辑、业务操作、危险操作”的顺序。

### 先迁移代表性核心页面

优先覆盖 system、operations、process、lowcode 中重复度高的列表页，并保留定制页面后续迁移空间。组件和规范先稳定，再扩大覆盖面。

## Risks / Trade-offs

- [风险] slot 组件仍允许调用方放入错误语义按钮 → 通过组件测试、页面测试和开发文档约束顺序。
- [风险] 批量迁移可能影响局部 flex 布局 → 保持组件为透明轻量容器，并执行类型检查与相关测试。
- [风险] 行操作过多时仍可能拥挤 → 组件提供换行/紧凑布局，本次不改变业务操作可见性，后续按页面引入“更多”。
- [取舍] 不做全仓一次性迁移 → 先覆盖主要业务页并提供唯一推荐组件，降低大范围视觉回归风险。

## Migration Plan

1. 新增公共布局组件、导出入口和组件测试。
2. 迁移用户、租户、字典等系统页。
3. 迁移公告、文件、消息、流程包和低代码表单等列表页。
4. 运行定向测试、类型检查与 OpenSpec 校验。
5. 如出现布局回归，可逐页撤回组件包装；业务逻辑不受影响。
