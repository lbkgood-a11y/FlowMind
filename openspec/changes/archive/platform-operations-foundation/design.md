## Context

平台已完成 RBAC、菜单/按钮授权、组织与数据权限、审计日志、字典、系统配置和会话管理。当前管理端还缺少企业中后台常见的运营支撑能力：公告通知、站内消息、文件中心、导入导出任务和后台任务调度。

这些能力具有明显的平台基础属性，后续业务微服务都可能复用，但不应该把业务数据继续堆叠到认证服务中。认证服务仍作为权限元数据来源，运营服务负责运营域数据和执行记录，各业务服务通过统一 API 和权限模型接入。

## Goals / Non-Goals

**Goals:**

- 提供一套可复用的运营支撑基础服务，满足中后台日常管理需求。
- 保持菜单、接口、按钮权限由 `service-auth` 统一维护，避免各微服务形成孤立权限体系。
- 新增 `service-ops` 承载运营业务数据，并通过网关暴露 REST API。
- 文件、导入导出、后台任务都必须具备可追踪记录，便于审计和排障。
- 前端页面必须使用与用户/角色/菜单管理一致的按钮授权逻辑。

**Non-Goals:**

- 不在本次实现多租户计费、复杂审批流或工作流编排。
- 不引入外部对象存储、分布式锁或独立调度集群作为强依赖。
- 不实现面向终端用户的实时 WebSocket 推送，第一版以轮询/查询站内消息为主。
- 不实现复杂导入模板 DSL，第一版提供任务模型和通用结果追踪。

## Decisions

### Decision 1: 新增 `service-ops` 承载运营域数据

`service-auth` 已经负责认证、授权、会话和部分治理能力，但通知、文件、导入导出和任务调度属于平台运营域。新增 `service-ops` 可以让运营能力成为独立微服务，后续被业务服务复用。

Alternatives considered:

- 继续放入 `service-auth`：实现更快，但认证服务职责会膨胀，后续拆分成本高。
- 放入每个业务服务：局部自治强，但文件、消息和异步任务模型会重复建设，权限口径难统一。

### Decision 2: 权限元数据仍由 `service-auth` 统一种子化

菜单、接口权限和按钮权限的生命周期继续归属认证服务。`service-ops` 的接口通过统一权限编码接入，前端按钮也使用相同编码控制显示与操作。

Alternatives considered:

- 运营服务维护自己的权限表：短期解耦，但会破坏“所有服务同一套权限体系”的要求。
- 前端写死按钮权限：实现简单，但授权变更不可治理，容易出现前后端权限不一致。

### Decision 3: 文件中心第一版使用本地存储适配器

文件元数据入库，物理文件使用本地目录存储，并通过配置项指定根路径。服务层使用存储适配器接口隔离实现，后续可替换为 MinIO、S3 或企业对象存储。

Alternatives considered:

- 直接引入对象存储：更接近生产，但增加本地开发部署复杂度。
- 只存数据库 BLOB：部署简单，但不利于大文件、下载性能和后续对象存储迁移。

### Decision 4: 导入导出以异步任务记录为核心

导入导出任务统一记录任务类型、业务编码、状态、进度、请求参数、结果文件和失败信息。第一版提供任务创建、查询、取消和结果下载能力，业务处理器可后续按业务域接入。

Alternatives considered:

- 同步导入导出：接口简单，但不适合大数据量和失败追踪。
- 每个页面独立实现导入导出：上线快，但缺少统一进度、结果和审计能力。

### Decision 5: 后台任务调度先实现单服务调度模型

第一版提供任务定义、启停、手动触发、执行日志和基础 cron 配置。运行时使用 Spring TaskScheduler 管理本服务内任务，执行结果入库。后续如需多实例严格去重，可引入 ShedLock、Quartz 集群或 Temporal Workflow。

Alternatives considered:

- 立即引入 Quartz 集群：能力完整，但当前基础平台阶段引入成本偏高。
- 只做手动触发：实现最小，但无法覆盖后台任务调度的核心管理需求。

## Risks / Trade-offs

- [Risk] 本地文件存储不适合多实例生产部署 -> Mitigation: 存储访问通过接口封装，配置项明确标注本地模式，后续可切换对象存储。
- [Risk] 单服务调度在多实例下可能重复执行 -> Mitigation: 第一版记录运行实例和执行日志，生产多实例启用前补充分布式锁或迁移 Temporal/Quartz。
- [Risk] 运营服务与认证服务权限种子存在部署顺序依赖 -> Mitigation: 权限数据仅作为授权元数据，不阻塞运营服务表结构创建；网关和前端发布时按后端先行。
- [Risk] 导入导出任务处理器没有统一业务扩展点会导致重复代码 -> Mitigation: 第一版定义任务模型和 API，服务内预留 processor 注册接口。

## Migration Plan

1. 新增 `service-ops` 模块、数据库迁移和基础配置。
2. 在 `service-auth` 迁移中种子化运营菜单、接口权限和按钮权限。
3. 网关增加 `/api/v1/announcements/**`、`/api/v1/messages/**`、`/api/v1/files/**`、`/api/v1/import-export-tasks/**`、`/api/v1/jobs/**` 路由。
4. 前端增加运营支撑页面并绑定按钮权限。
5. 本地验证后端测试、前端类型检查、OpenSpec 校验和关键接口流程。

Rollback:

- 可先从网关移除运营路由和前端菜单入口，保留数据库表不影响既有功能。
- 如需完全回滚，删除 `service-ops` 模块发布物并执行数据库回滚脚本或保留空表待后续复用。

## Open Questions

- 生产文件存储最终选择 MinIO、S3 还是企业已有对象存储，需要在部署环境明确。
- 导入导出任务的第一批业务处理器是否优先接入用户、角色、菜单，还是先提供平台通用任务壳。
- 后台任务调度是否需要跨实例强一致去重，取决于第一阶段部署是否多实例运行。
