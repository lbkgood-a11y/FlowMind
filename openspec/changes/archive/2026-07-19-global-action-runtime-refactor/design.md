## Context

TrioBase already has several operation-like models:

- `service-lowcode` defines application actions and a runtime action endpoint for form save, submit, workflow launch, and workflow retry.
- `service-workflow-engine` defines process launch, task operations, business action executors, closure effects, process outcomes, and closure retry/manual handling.
- `service-openapi` defines governed integration invocation and orchestration execution.
- `service-auth` and common modules provide permission checks, security context, request audit, and TraceId propagation.
- `apps/web-antd` currently calls operation-specific APIs from each page or dialog.

These local models solve their immediate use cases, but they duplicate the same operation concerns: request shape, action naming, confirmation, authorization, guard checks, idempotency, status mapping, retryability, result semantics, trace correlation, and audit visibility. The Global Action Runtime refactor turns these into a shared platform contract while preserving service ownership of business state and Temporal workers.

## Goals / Non-Goals

**Goals:**

- Define one Global Action envelope and result lifecycle for all business-changing operations.
- Route GUI, LUI, Agent, scheduler, event, and API sources through the same action contract.
- Persist action execution state so operation status is queryable and auditable by `actionId`, `correlationId`, `traceId`, `idempotencyKey`, actor, source, target, and status.
- Centralize common validation, authorization, state guard, idempotency, execution routing, event, and audit flow.
- Keep business state changes inside the owning service and keep Temporal workers embedded in their Spring Boot services.
- Remove obsolete local action DTOs, duplicated frontend business API wrappers, local status translations, and redundant permission/idempotency/audit code after migration.
- Support action-level SSE/polling so frontend pages and future LUI panels can show the same operation lifecycle.

**Non-Goals:**

- Do not model local UI events such as opening a drawer, switching tabs, refreshing a table, or local input edits as Global Actions.
- Do not move all business data ownership into a central action service.
- Do not allow arbitrary script, SQL, URL, dynamic class names, dynamic components, or free Prompt execution through action definitions.
- Do not use Global Action as a replacement for Domain Events; Action means requested operation, Event means fact already produced.
- Do not bypass existing Temporal determinism, Activity idempotency, TraceId, AI gateway, or authorization guard rules.

## Decisions

### 1. Global Action Is The Mandatory Business Operation Contract

All operations that mutate business state, trigger workflow, invoke external side effects, require retry/compensation, or require audit MUST be represented as a `GlobalActionRequest`.

The envelope contains:

- `actionId`: generated by the runtime unless supplied for trusted internal replay.
- `actionType`: namespaced action code, for example `lowcode.form.submit`, `process.instance.start`, `process.task.approve`, `process.closure.effect.retry`, `integration.orchestration.start`.
- `source`: `GUI`, `LUI`, `AGENT`, `API`, `EVENT`, `SCHEDULER`, or `WORKFLOW`.
- `actor`: user, agent, service, system, or scheduler identity.
- `target`: target type, target id, owner service, tenant id, and optional version.
- `payload`: schema-validated action parameters.
- `context`: trace id, request id, correlation id, locale, auth/policy versions, and optional confirmation metadata.
- `idempotencyKey`: required for state-changing and retryable operations.
- `executionMode`: `SYNC`, `ASYNC`, `WORKFLOW`, or `SIGNAL`.

Alternative considered: keep local DTOs and only add shared helper methods. This was rejected because LUI/Agent execution and audit correlation would still need to understand every local DTO and status model.

### 2. Use A Hybrid Runtime: Central Action Facade, Owner-Service Executors

Introduce a new `service-action` module as the central Action facade for public submission, action status, event stream, and audit lookup. The facade owns action identity, registry lookup, status records, idempotency keys, and action events.

Business execution remains owned by the service that owns the target resource:

- `service-lowcode` owns form and lowcode runtime executors.
- `service-workflow-engine` owns process launch, task signal, closure effect, and business closure executors.
- `service-openapi` owns integration invocation and orchestration executors.
- `service-auth` owns authorization decisions, but does not own domain guard checks.

Owner services register `ActionDefinition` and `ActionExecutor` implementations through a shared action runtime library. The central facade dispatches to owner adapters according to the registered action type and execution mode. Long-running or cross-service operations use durable action events, outbox, Temporal workflow start/signal, or owner-managed asynchronous execution rather than ad hoc chained synchronous Feign calls.

Alternative considered: put all action execution in one central service. This was rejected because it would violate service ownership, grow into a business rule monolith, and conflict with Temporal worker colocation rules.

Alternative considered: only embed action endpoints in each owner service without a facade. This was rejected because the frontend, LUI, Agent, and audit views need a single place to submit and observe action state.

### 3. Action Definitions Are Code-Registered And Metadata-Described

Each executable action has a code-registered `ActionDefinition`:

- `actionType`
- `ownerService`
- `targetType`
- `payloadSchema`
- `requiredPermission`
- `requiredGuards`
- `allowedStates`
- `executionMode`
- `requiresConfirmation`
- `auditLevel`
- `sensitivePayloadPaths`
- `resultSchema`
- `retryPolicy`
- `compensationActionType`

Database or published metadata can describe labels, ordering, availability, default confirmation copy, and binding to lowcode/process catalog entries, but it MUST NOT define arbitrary executable code or unregistered executor keys. This keeps configurability without handing execution authority to mutable metadata.

### 4. Action Status Is The Cross-Module Operation Lifecycle

Global status values are:

- `CREATED`
- `VALIDATING`
- `REJECTED`
- `AUTHORIZED`
- `ACCEPTED`
- `RUNNING`
- `SUCCEEDED`
- `FAILED`
- `CANCELLED`
- `COMPENSATING`
- `COMPENSATED`

Domain-specific states remain in `result.data` or target resources. For example, a lowcode form can still store `PENDING_WORKFLOW`, a process instance can still be `RUNNING`, and a closure effect can still be `RETRYING`, but the action result visible to clients uses the global lifecycle.

Alternative considered: reuse the richest existing status model, such as closure effect status or OpenAPI execution state. This was rejected because those states describe domain execution details, not a universal operation request lifecycle.

### 5. Idempotency Is Enforced Before Execution And Again In Owner Services

The central facade enforces `(tenantId, actionType, idempotencyKey)` uniqueness for state-changing actions and returns the existing action result for duplicate submissions. Owner services keep their existing business-level idempotency constraints such as process launch keys, task operation ids, closure effect ids, and external integration invocation keys.

This two-layer model prevents duplicate dispatch and still protects against retries or partial failures inside the owner service.

### 6. Authorization Uses Action Context Plus Owner Guards

The runtime first calls the enterprise authorization decision API using action-level resource/action context. Owner services then evaluate local guards such as workflow task candidate, no-self-approval, document status, closure retry availability, integration capacity, or OpenAPI release state.

The final action decision is allowed only if both central authorization and owner guards allow the operation. Denials are recorded with bounded, non-sensitive reason details.

### 7. Frontend Uses `dispatchAction` For Business Operations

`apps/web-antd` gets one Action Client and dispatch composable. Business pages no longer call operation-specific mutation APIs directly. They dispatch Global Actions and react to the normalized result lifecycle:

- show confirmation when required
- submit action
- show `ACCEPTED` or `RUNNING`
- wait via SSE or polling when needed
- refresh affected queries on `SUCCEEDED`
- show structured errors on `REJECTED` or `FAILED`

Local UI events remain local component state and never enter the Global Action Runtime.

### 8. LUI And Agent Tool Calls Are Action Candidates Until Confirmed

LUI intent parsing and Agent tool calls produce `ActionCandidate` objects, not direct mutations. A candidate must match a registered `ActionDefinition`, pass schema validation, pass authorization, and include confirmation metadata when the action definition requires human confirmation. Only then can it become a Global Action.

This preserves frontend rule 10 and rule 11: generated UI uses registered components and schema validation, and LUI changes GUI state through the action/state flow rather than DOM mutation.

## Risks / Trade-offs

- [Large blast radius] -> Split migration by operation family, keep temporary compatibility adapters, and delete old paths only after tests and UI migration are complete.
- [Central facade becomes a bottleneck] -> Keep owner execution in owner services, index action records carefully, and dispatch long-running work asynchronously.
- [Duplicate idempotency semantics] -> Define central idempotency rules and map existing business keys into `idempotencyKey` before deleting local operation id concepts.
- [Status mapping loses useful detail] -> Keep domain states in target resources and action result details; only the action envelope lifecycle is normalized.
- [Audit payload may expose sensitive fields] -> Define `sensitivePayloadPaths`, store redacted payload summaries, and never persist raw prompt or secret values in action audit.
- [Breaking frontend churn] -> Migrate through an Action Client facade so pages change one operation at a time without rewriting table/detail rendering.
- [Temporal misuse risk] -> Keep workflow I/O inside Activities, use existing trace context propagation, and route workflow-starting/signaling actions through owner workflow services.

## Migration Plan

1. Add shared action DTOs, enums, errors, and schema contracts in common modules.
2. Add action execution, event, and audit persistence with indexes and redaction support.
3. Add the `service-action` module with central Action facade endpoints for submit, status, result, SSE events, and query.
4. Add owner-service action runtime library support and register initial lowcode, workflow, closure, and integration action definitions.
5. Migrate lowcode runtime action execution and remove `RuntimeActionRequest`/`RuntimeActionResponse` after frontend migration.
6. Migrate process instance start, task approve/reject/transfer/add-sign, closure retry/manual-handled, and OpenAPI runtime execution to Global Action.
7. Add frontend Action Client and replace direct business-changing API wrappers in `apps/web-antd`.
8. Add LUI/Agent candidate-action bridge and block unregistered or unconfirmed tool calls.
9. Update audit views, diagnostics, tests, ArchUnit, and frontend lint/test coverage.
10. Remove compatibility endpoints, obsolete DTOs, duplicate status utilities, and unused API wrappers.

Rollback strategy: before compatibility endpoints are removed, old endpoints remain as adapters that submit equivalent Global Actions. If action runtime causes a production issue during migration, route affected operation families back to their adapter path while preserving action audit entries. After compatibility deletion, rollback requires restoring the previous release and database-compatible migrations.

## Open Questions

- No blocking open questions remain for implementation. The first implementation uses a new `service-action` module, supports both status polling and SSE action events, and keeps compatibility adapters only until each operation family is migrated and tested.
