## Context

`service-lowcode` 已有表单版本、应用版本、页面与动作元数据、发布授权同步和通用运行时后端；web-antd 只有表单管理和运行中心。auth migration V50 注册了 `/lowcode/application` 与 `/lowcode/application/list`，但对应 Vue 页面不存在，形成 404。现有页面元数据主要是自由 JSON，缺少可视化列表设计与跨表单关系模型。

## Goals / Non-Goals

**Goals:**

- 交付可操作的应用管理工作台，复用现有应用生命周期 API。
- 让发布表单能够被应用选择、设计列表/详情/创建页面、配置动作并发布。
- 用结构化、白名单元数据描述列表页，不执行任意代码。
- 支持最大三级的有向无环主—子—孙表单关系，以及级联实例数据。
- 发布时固化关系和页面快照，运行时与授权决策使用同一版本。
- 保持现有单表表单和费用报销兼容入口可用。

**Non-Goals:**

- 本期不支持四级以上层级、跨租户关系或多主表聚合。
- 不提供任意 SQL、JavaScript、表达式执行或自定义 Vue 组件注入。
- 不建设通用 BI 查询器；列表只针对应用已绑定表单实例。
- 不在本变更中建设物理多租户隔离。

## Decisions

### 关系模型独立于 JSON Schema

新增 `lc_form_relation`，记录父/子表单版本、关系编码、基数、父键、子外键、排序和级联策略。关系属于应用版本草稿并在发布时固化。相比把关系塞入 JSON Schema，这让引用完整性、版本校验、索引和授权同步可独立治理。

### 限制为三级有向无环图

发布校验执行 DFS：禁止环、自引用、重复 relation code、深度超过 3、跨租户引用、未发布表单引用以及不存在/类型不兼容的关联字段。三级满足主单—明细—明细扩展的企业常见场景，同时控制运行时复杂度。

### 级联数据使用独立关系实例表

保持现有 `lc_form_instance` 每个节点独立存储，并新增 `lc_form_instance_relation` 保存父子实例关联、关系编码和序号。级联提交在单个低代码服务事务中先校验全部节点，再写主、子、孙实例及关系；任一失败整体回滚。

### 列表设计使用版本化结构元数据

LIST 页面 `metadataJson` 采用受控结构：`columns`、`filters`、`defaultSort`、`rowActions`、`pageSize`。后端解析并验证字段引用、格式枚举、宽度和操作引用；前端提供图形化编辑器并输出同一结构。

### 应用管理与运行中心职责分离

“应用管理”面向设计者，支持 CRUD、派生版本、编排、发布、下线；“应用中心”面向最终用户，只列出已发布且 `LOWCODE_APP:<APP_KEY>:VIEW` 决策允许的应用。动态菜单组件修正为真实存在的路径。

### 授权分为应用入口和表单业务两层

应用发布同步 `LOWCODE_APP` 的 VIEW/DESIGN/PUBLISH/OFFLINE；所有引用表单已在各自发布时同步 `LOWCODE_FORM` 动作与字段。角色至少需要应用 VIEW 和表单 VIEW；创建、编辑、提交、审批按动作分别授权。菜单可见性继续由 grant 投影，不新增第二授权源。

## Risks / Trade-offs

- [风险] 级联提交载荷变大且校验复杂 → 最大三级、限制节点数、单事务预校验并建立关系索引。
- [风险] 已有页面 metadataJson 结构不一致 → 解析器提供兼容默认值，旧应用按默认列表列运行。
- [风险] 应用发布后表单新版本出现 → 发布快照固定表单 definition id；应用必须派生新版本才能升级。
- [风险] 应用可见但表单动作不足 → 运行时批量决策并隐藏/禁用动作，管理端发布前给出授权提示。
- [风险] 菜单迁移与静态路由重复 → 动态菜单路径对齐现有 Vue 组件，路由名称保持唯一。

## Migration Plan

1. 增加关系表与实例关系表迁移，不修改已有实例数据。
2. 增加 auth 菜单修复迁移，将 M033 组件指向真实应用管理页，将 M035 对齐应用中心。
3. 扩展后端 DTO、校验、CRUD、发布与运行时 API。
4. 上线应用管理、列表设计和关系设计前端。
5. 对已有发布应用使用兼容元数据；费用报销进行回归验证。
6. 验证角色授权、应用中心可见性和级联提交。

回滚时保留迁移表，隐藏新菜单并回退前端；旧单表运行时不依赖新关系数据。
