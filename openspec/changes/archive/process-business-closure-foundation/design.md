## Context

当前流程运行时 MVP 已经覆盖流程包版本、表单快照、参与者解析、候选任务、条件分支、审批/驳回/转办/加签、审计和 Temporal 恢复语义。但这个闭环主要是“流程状态闭环”：流程能从发起走到终态，并不能证明对应业务对象已经完成状态回写、事件投递、通知、数据沉淀或 Agent 后续处理。

TrioBase 的目标是兼顾业务流程、数据驱动和 AI/Agent 三个应用方向，因此流程平台需要一层业务对象驱动的闭环能力：业务人员通过业务对象、表单、权限、单据状态和闭环动作完成选择式配置；系统在发布时编译稳定计划；运行时按计划高效、幂等、可追踪地执行。

本设计继续遵守现有工程铁律：Temporal Workflow 保持确定性，I/O 和业务副作用必须在 Activity 或后台执行器中完成；Activity 和 effect 必须幂等；TraceId 必须贯通；设计器不得暴露任意脚本、URL、SQL 或无沙箱动态结构。

## Goals / Non-Goals

**Goals:**

- 在 `service-workflow-engine` 内建立第一版业务对象目录，支持数据库注册业务对象、状态、表单、权限、动作、事件和 Agent follow-up 动作。
- 使用报销单跑通业务对象目录、两种发起模式、权限协同、流程审批、状态回写、事件、通知、Agent follow-up 和失败重试。
- 让流程设计器以业务对象为起点，通过选择器、模板、业务语言提示和四类可视化完成配置。
- 支持已有单据发起和新建单据并发起，并在发起时执行受控 StartEffect。
- 建立 `ProcessOutcome`、`ClosureRecord`、`ClosurePlan`、effect 执行记录和 Outbox，支持硬/软闭环。
- 通过发布快照隔离目录、表单、权限和闭环配置的后续变更。

**Non-Goals:**

- 不实现任意 `SERVICE_TASK` 编排、通用 Saga、子流程或运行中实例迁移。
- 不允许配置人员填写任意 URL、SQL、脚本、动态类名、JSON Path、idempotency template 或自由 Prompt。
- 不做业务服务自注册协议；第一版目录由数据库维护，副作用执行器由代码注册。
- 不把流程引擎改造成报销、采购、工单等业务系统；真实业务事实仍归业务服务所有。
- 不开放底层 JSON/编码编辑能力；第一版仅提供只读技术预览。

## Decisions

### 1. 业务对象目录先放在流程平台内

`service-workflow-engine` 新增业务对象目录表和查询 API，作为流程、表单、权限、单据状态、闭环动作和 Agent follow-up 的统一选择项来源。第一版不抽独立服务，避免在核心模型未稳定前扩大服务边界；后续若多个平台模块复用，可以再抽为公共目录能力。

业务对象目录支持 `GLOBAL + tenant override`。全局目录提供标准对象和模板，租户可以覆盖展示名、可选状态、表单或默认闭环策略；发布流程时使用解析后的有效目录快照。

### 2. 数据库注册目录，代码注册执行器

数据库只注册“能选什么”：业务对象、状态、表单、权限动作、业务动作、事件、Agent follow-up 动作、参数 Schema 和展示文案。所有会产生副作用的执行能力必须由代码注册，并通过 `executorKey` 与目录项绑定。

如果目录中配置了 `executorKey` 但应用中没有对应执行器，启动阶段记录告警和健康检查诊断，流程发布阶段强制失败。这样平台人员可以通过数据库维护目录，配置人员可以选择业务选项，开发人员仍控制真正的副作用边界。

### 3. 目录也有生命周期

业务对象目录记录使用 `DRAFT / PUBLISHED / OFFLINE` 状态。设计器可以查看草稿目录，但流程发布只能引用已发布目录。目录下线不得影响已经发布的流程版本，因为流程发布时会固化业务对象、状态、权限、表单和闭环计划快照。

### 4. 流程以业务对象驱动配置

新建流程时必须先选择已发布业务对象。选择后，设计器从目录中加载可选表单、业务状态、权限动作、发起方式、闭环动作、事件、通知对象和 Agent follow-up 动作。配置人员看到的是“报销单、已通过、提交报销、通知申请人”等业务语言，而不是 `businessRef`、`connector/action` 或权限编码。

条件分支、参与人、闭环参数也采用选择优先原则。默认使用字段选择器和条件构造器，例如“当 报销金额 大于 5000 时进入 财务审批”；高级表达式可作为后续管理员能力，不进入第一版。

### 5. 兼容两种发起模式

已有单据发起：从业务单据详情页或 API 传入 businessRef，流程运行时校验用户权限、单据状态和发起策略，再执行 StartEffect，例如把报销单从“草稿”更新为“审批中”。

新建单据并发起：用户填写发起表单，流程运行时调用代码注册的创建单据执行器生成 businessRef，再执行后续 StartEffect 和启动流程。创建动作必须是注册执行器，不能由目录配置任意接口。

### 6. 全闭环区分硬闭环和软闭环

StartEffect 和关键 OutcomeEffect 可以配置为硬闭环：失败时不得静默展示业务完成，而是阻断发起、进入 `CLOSURE_FAILED` 或保持可诊断挂起状态。通知、领域事件和 Agent follow-up 默认软闭环：失败不会回滚审批结果，但会把 `ClosureRecord` 或 effect 标记为失败并允许授权人员重试。

每个 effect 必须具备稳定幂等键，推荐由流程实例、业务对象、结果状态、effect key 和目录快照版本组成。重复投递 `ProcessOutcome` 或重复重试 effect 不得产生重复业务副作用。

### 7. ProcessOutcome 是三方消费契约

流程进入业务结果态时生成唯一 `ProcessOutcome`，包含 `processRef`、`businessRef`、`outcome`、`actorContext` 和 `traceId`。业务回写、数据沉淀和 Agent follow-up 都消费 Outcome/Closure 契约，不直接依赖 Workflow 内存状态或内部任务表。

### 8. 发布时编译 ClosurePlan

流程包发布时校验业务对象目录、表单字段、权限动作、状态转换、effect 参数 Schema、executorKey、Agent 动作和硬/软闭环规则，并编译不可变快照。运行时只读取快照后的 `LaunchPlan`、`PermissionPlan`、`ClosurePlan` 和 `AgentFollowUpPlan`，不重新解释设计器草稿。

### 9. 设计器必须完整可视化

第一版设计器必须提供四类可视化：

- 流程图：节点、连线、参与人、条件分支。
- 单据状态图：允许发起状态、发起后状态、通过/驳回/失败后的业务状态。
- 闭环动作链：每个 outcome 触发哪些业务动作、事件、通知和 Agent follow-up。
- 权限矩阵：谁能发起、查看、办理、重试闭环和触发 Agent follow-up。

底层技术结构可只读预览，但不得作为完成配置的主要入口。

### 10. 权限绑定现有 RBAC

业务对象目录维护 `actionCode -> permissionCode` 的映射，`permissionCode` 直接绑定现有 RBAC 权限编码。流程发起、查看、办理、闭环重试和 Agent follow-up 都必须同时满足流程规则、业务对象权限和任务候选/受理人规则。

### 11. Agent follow-up 只允许预注册动作

Agent follow-up 是受控 effect 类型。目录中注册可选 Agent 动作、展示名、参数 Schema、授权动作和 executorKey；设计器只提供选择和参数表单，不开放自由 Prompt 或任意工具调用 JSON。运行时通过已注册执行器触发 Agent 编排，并把结果写入 effect 记录和 Closure 状态。

### 12. 失败处理以重试为主，人工标记必须审计

第一版提供失败 effect 的授权重试。若需要“人工标记已处理”，必须作为审计动作落库，记录操作人、原因、TraceId、原始失败信息和最终处置结果，不能直接抹平失败记录。

## Risks / Trade-offs

- [目录放在流程平台内可能形成平台膨胀] → 第一版只承载流程配置所需元数据，不承载业务数据和业务规则执行；后续达到跨模块复用时再拆公共目录。
- [数据库目录与代码执行器不一致] → 启动诊断告警，发布强校验，CI 加入 fixture 校验，避免运行时才暴露缺失。
- [硬闭环增加用户等待和失败路径复杂度] → 仅允许 StartEffect 和关键 OutcomeEffect 使用硬闭环，通知、事件、Agent 默认软闭环。
- [目录变更与已发布流程版本不一致] → 发布快照固化目录和计划，实例始终执行启动时版本。
- [设计器选择项过多导致复杂] → 以业务对象模板、推荐默认值、完成度检查和业务语言预览降低配置负担。
- [Agent follow-up 可能引入不可控行为] → 仅允许预注册动作、参数 Schema、授权校验和效果审计，不开放自由 Prompt。
- [重试可能重复副作用] → 每个 effect 必须使用业务幂等键，执行器必须处理重复请求。

## Migration Plan

1. 增加业务对象目录和闭环记录相关数据库迁移，使用 Flyway seed 注册报销单目录、状态、表单引用、权限动作、闭环动作、事件和 Agent follow-up 示例。
2. 在 `service-workflow-engine` 增加目录读取、发布校验、执行器注册和诊断能力；先不暴露目录管理 UI。
3. 扩展流程包发布快照，兼容未配置业务对象的旧流程；旧流程可继续作为纯流程闭环运行，但不能执行业务回写。
4. 增加 Outcome/Closure/effect/outbox 运行时表和后台执行器，先对费用报销 fixture 启用。
5. 升级设计器为业务对象向导，保留现有 JSON 只读预览，逐步减少 JSON textarea 作为主流程入口。
6. 增加实例详情闭环状态、effect 明细、失败原因和重试入口。
7. 回滚时应用可回退到旧流程运行时；新增目录和闭环表保留，不影响旧流程包执行。

## Open Questions

当前没有阻断本 change 的开放问题。目录独立服务化、业务服务自注册协议、高级表达式模式和通用服务编排留待后续 change 决定。
