## Context

TrioBase 当前已具备基础认证、RBAC 菜单按钮权限、组织维度和数据权限策略。下一步需要补齐企业中后台的治理能力，让平台能回答“谁在什么时候做了什么”“哪些会话正在使用系统”“平台枚举从哪里来”“系统策略如何调整”。这些能力属于平台横切层，应优先放在 `service-auth` 内，复用现有权限、菜单、网关和前端系统管理入口。

## Goals / Non-Goals

**Goals:**

- 提供操作审计日志查询能力，并在管理 API 执行时记录用户、权限、请求、响应结果、耗时和 TraceId。
- 提供登录日志和会话管理能力，支持登录成功/失败记录、当前会话查询、退出和管理员强制失效。
- 提供数据字典类型和字典项管理，作为前端枚举和后端校验的统一来源。
- 提供系统参数配置管理，支持平台策略项的查看、更新、脱敏展示和缓存友好读取。
- 将治理页面接入“系统管理”，按钮权限继续沿用 API 权限码模型。

**Non-Goals:**

- 不在第一版实现分布式在线状态强一致统计；会话以数据库记录和 JWT `jti` 关联为主。
- 不在第一版实现复杂审计 diff 引擎；先记录请求摘要、资源 ID、结果和错误信息，预留 before/after 字段。
- 不实现灰度配置中心或 Nacos 动态配置替代方案；系统参数只管理业务平台参数。
- 不做外部 SIEM、ELK、审计归档集成；只提供平台内查询和后续导出的基础。

## Decisions

### D1: 治理能力先归属 `service-auth`

操作审计、登录日志、会话、字典和系统参数都先放在 `service-auth`。理由是这些能力强依赖当前用户、角色、权限和菜单种子，并且第一版主要服务系统管理后台。

备选方案是新建 `service-governance`。该方案更干净，但会立即引入跨服务鉴权、菜单种子、部署和网关路由复杂度；当前阶段先保持实现集中，后续如治理能力变重再拆分。

### D2: 审计记录采用过滤器 + 注解元数据混合

现有 `AuditSecurityFilter` 已能接入请求上下文。第一版在请求结束后按管理 API、用户上下文和权限元数据记录审计日志。Controller 不需要逐个手写日志，特殊场景再通过后续注解扩展动作名和资源 ID。

### D3: 会话使用 JWT `jti` 做关联键

登录成功时记录 access token 的 `jti`、用户、过期时间、客户端摘要和状态。退出或管理员强制失效时标记会话状态。网关和服务端后续可按 `jti` 黑名单/状态表校验主动失效。

第一版先实现记录、查询和标记失效；不要求所有网关请求立即查询会话表，避免每次请求增加数据库压力。

### D4: 数据字典分类型表和字典项表

`sys_dict_type` 维护字典编码、名称、状态、是否系统内置；`sys_dict_item` 维护 label/value/sort/status/tag/metadata。前端需要枚举时可按 `dictCode` 查询启用项，管理页面可维护完整数据。

### D5: 系统参数支持敏感值脱敏展示

`sys_config` 保存配置键、值、类型、分组、是否敏感、是否内置和描述。敏感参数管理列表只返回脱敏值，更新时可以覆盖真实值。读取接口提供按 key 查询和按前缀查询，后续可以接 Redis 缓存。

### D6: 权限码仍以 API 权限为唯一控制面

页面菜单挂 GET 权限，新增/修改/删除/强制失效等动作挂对应 POST/PUT/DELETE 权限。搜索、刷新、分页和详情继续复用页面 GET 权限，不额外创建搜索按钮权限。

## Risks / Trade-offs

| Risk | Mitigation |
|------|------------|
| 审计日志量增长较快 | 第一版按系统管理 API 记录，保留分页查询；后续增加归档和按模块采样 |
| 会话主动失效没有立即接入网关校验 | 先记录和管理状态，后续将 `jti` 状态检查接入网关缓存 |
| 字典被误改影响前端显示 | 系统内置字典支持 `systemFlag`，删除受限，优先允许禁用和修改显示 |
| 系统参数误配置影响登录/权限 | 内置参数保留默认值和类型校验，敏感项脱敏展示 |
| 治理能力放在 auth 服务后续可能变重 | 表和 API 保持平台治理边界清晰，后续可按能力拆出独立服务 |

## Migration Plan

1. 新增 Flyway 迁移创建审计、登录日志、会话、字典、系统参数表。
2. 初始化系统管理下的治理菜单、按钮、API 权限和管理员授权。
3. 初始化基础字典类型和系统参数默认值。
4. 后端实现管理 API，并在认证/审计过滤器中写入日志。
5. 前端新增治理页面并接入系统管理路由。
6. 回滚时可禁用新增菜单；新增表为增量表，不影响现有认证和权限主链路。

## Open Questions

第一版采用以下默认口径：

1. 审计日志只在平台内保留，不对接外部日志系统。
2. 会话强制失效先标记状态，网关强校验作为后续增强。
3. 字典和系统参数先按默认租户 `default` 实现，保留 `tenant_id` 字段。
