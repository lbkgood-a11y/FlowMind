## Context

当前菜单管理使用 Ant Design Vue 树形 Table 展示 `sys_menu` 全量节点，并在同一层级混合目录、菜单、按钮、内嵌页面、外链和隐藏 API 权限。页面依赖横向滚动承载权限码、路由、组件和操作列；编辑通过 790px 抽屉完成，但字段按数据结构排列，排序与分组字段还被 CSS 隐藏。

本次改版面向平台管理员，核心任务是理解和维护导航结构，其次才是配置路由与按钮权限。现有菜单 CRUD API、动态路由协议和 `sys_menu` 数据结构可以继续使用，不需要后端模型迁移。

## Goals / Non-Goals

**Goals:**

- 让导航层级成为页面的第一视觉和第一操作对象。
- 将按钮/API 权限从主导航树中移出，作为所属业务菜单的附属配置。
- 建立可预测的默认展开、全部展开、恢复收起和搜索状态恢复行为。
- 降低误编辑风险，并让不同菜单类型只暴露相关字段。
- 支持同级排序和明确的跨层级移动方式。
- 保持既有菜单接口、权限模型和动态路由兼容。

**Non-Goals:**

- 不修改 `sys_menu`、`sys_role_menu` 或 `sys_permission` 数据模型。
- 不重写动态路由生成机制。
- 不在首版实现跨层级拖拽、批量删除或版本历史。
- 不改变角色授权页面的职责。

## Decisions

### 1. 使用左右主从布局

页面主体采用固定宽度左栏和自适应右栏：

```text
┌──────────────────────────────────────────────────────────────┐
│ 菜单管理     搜索 / 类型 / 状态             展开切换  新增  │
├──────────────────────┬───────────────────────────────────────┤
│ 导航结构             │ 节点详情                              │
│ ▾ 系统管理           │ 基本信息                              │
│   ▸ 用户管理         │ 路由配置                              │
│   ▸ 角色管理         │ 权限配置                              │
│ ▾ 快速开发           │ 显示设置                              │
│   ▸ 表单管理         │                         编辑 / 新增下级│
└──────────────────────┴───────────────────────────────────────┘
```

左栏使用全局统一的紧凑主栏宽度 280px，右栏占剩余空间；分栏间距统一使用公共变量。相比继续扩展树形 Table，该结构能避免横向滚动，并允许权限配置拥有独立信息密度。

备选方案是保留树形 Table 并减少列，但按钮/API 节点仍会持续污染树结构，因此不采用。

### 2. 在前端构建导航树和权限索引

页面加载全量菜单后构建两个派生结构：

- `navigationTree`：包含 `catalog/menu/link/embedded` 节点。
- `permissionChildrenByMenuId`：将 `button` 及满足隐藏 API 特征的节点按最近业务祖先归组。

仍保留原始 `allMenus` 作为编辑、父级选择和 CRUD 刷新的数据源。派生结构使用纯函数生成，便于单测覆盖，不引入新的全局状态库。

### 3. 展开状态采用三种显式状态

页面维护：

- `expandedKeys`：当前真实展开节点。
- `preSearchExpandedKeys`：进入搜索前快照。
- `isFullyExpanded`：由全部可展开节点是否都包含在 `expandedKeys` 中派生，而不是单独维护可能失真的布尔值。

首次加载只取第一个存在子节点的根节点作为默认展开键。点击“收起全部”调用同一默认展开计算函数，因此行为始终回到“仅第一个一级目录展开”，而不是完全折叠。

### 9. 统一紧凑分栏与查询操作规范

全局紧凑样式提供主栏宽度、分栏间距、查询标签宽度、查询控件高度和操作区对齐变量。菜单、角色、组织和数据权限主从页面必须共同引用该变量，不保留页面级固定宽度。查询输入框、选择框和日期控件使用一致的盒模型、精确高度和满列宽度，查询栅格使用等宽列；查询与页面操作按钮统一靠右。菜单导航面板高度由可视窗口计算，标题固定，树内容单独滚动。

### 4. 搜索使用临时派生树

搜索不直接覆写用户展开偏好。首次进入搜索状态时保存 `expandedKeys`；过滤结果保留匹配节点、祖先路径和必要子节点，并临时展开祖先。清空全部筛选条件时恢复快照。

### 5. 详情与编辑分离

右侧默认使用只读描述布局。点击编辑后切换为表单；取消恢复原始快照，保存成功后刷新数据并退出编辑状态。

当存在脏数据时，节点切换、关闭编辑和刷新操作都需要确认。该模式比选中即编辑更适合高权限治理页面。

### 6. 表单按节点类型和业务语义分区

编辑区域分为：

- 基本信息：类型、菜单标识、显示名称、上级菜单、状态、排序、分组。
- 路由配置：路由地址、激活路径、组件、链接地址。
- 权限配置：权限标识及所属按钮/API 权限列表。
- 显示设置：图标、激活图标、徽标、缓存、固定、隐藏行为。

字段可见性继续复用现有 `fieldVisibility` 思路，但由分区组件消费。删除固定“新增”标题附加内容。

### 7. 权限配置作为当前菜单的内嵌列表

右侧权限区域使用紧凑表格展示权限名称、权限标识、状态、排序和操作。新增权限默认 `parentId = selectedBusinessMenu.id`、`menuType = button`。隐藏 API 权限仍保留现有数据表达，只改变页面呈现方式。

### 8. 首版同级排序复用现有更新接口

拖拽完成后重新计算同一父级下的 `sortOrder`，使用有间隔的序号（10、20、30……），仅提交排序发生变化的节点。保存期间锁定该父级的再次拖拽，任一请求失败后重新加载服务端数据并提示用户。

批量排序接口延后评估；首版菜单规模下复用现有更新接口成本更低，也避免后端契约变更。

## Risks / Trade-offs

- [权限节点归属规则可能遇到历史脏数据] → 对找不到业务祖先的按钮/API 节点增加“未归类权限”分组，不静默丢弃。
- [逐项更新排序存在部分成功] → 使用间隔排序、串行提交和失败后强制刷新；后续菜单规模增长时再增加批量排序接口。
- [左右布局在小屏幕空间不足] → 低于 900px 时切换为树列表页与详情抽屉，不强行并排。
- [搜索和展开状态组合复杂] → 将树派生、默认展开和搜索恢复拆成纯函数并增加状态测试。
- [隐藏权限节点后管理员可能误以为数据缺失] → 在节点详情显示权限数量，并提供“未归类权限”入口及明确说明。

## Migration Plan

1. 在现有页面内先提取树构建、权限归组和展开状态纯函数并补测试。
2. 新增左侧导航树和右侧只读详情，不移除旧编辑逻辑。
3. 将编辑抽屉字段拆分为分组编辑组件，并接入右侧编辑态。
4. 将按钮/API 权限从主树隐藏并迁入权限配置区。
5. 接入搜索筛选、展开交替和同级拖拽排序。
6. 完成回归后移除旧树形 Table、列设置和全屏表格逻辑。

回滚时可恢复旧 `list.vue` 页面实现；本次不涉及数据库迁移，因此不存在数据回滚。

## Open Questions

- 当前“隐藏 API 权限”的识别是否只依赖 `menuType=button`，还是需要同时兼容历史 `hideInMenu=1` 且无路径的菜单节点；实施前应使用现网数据验证归组规则。
- 若同级菜单数量显著超过 100，是否需要在本次变更中提前增加后端批量排序接口。
