## Context

TrioBase 当前已有 `service-auth` 的用户、角色、菜单、API 权限体系，也已有 `service-org` 的基础组织树和用户组织关系雏形。现有 RBAC 能解决“能不能访问页面、按钮和接口”，但组织仍是单棵部门树，无法表达法人组织、行政组织、核算组织、业务组织等多视角结构，也无法统一支撑微服务里的数据权限过滤。

本变更在不推翻现有 RBAC 的前提下，引入组织维度、数据权限策略和统一权限执行基础。前端入口统一放在“系统管理”下，新增“组织管理”和“数据权限”两个页面。

## Goals / Non-Goals

**Goals:**

- 支持组织单元与组织维度分离，同一组织对象可出现在多棵组织树中。
- 支持 `ADMIN`、`LEGAL`、`ACCOUNTING`、`BUSINESS` 内置组织维度。
- 支持用户在不同组织维度下的归属维护，并保留主归属概念。
- 支持角色级数据权限策略，按资源、动作、组织维度、范围类型和组合方式描述数据范围。
- 支持前端维护组织树、用户组织归属和角色数据权限。
- 补齐组织管理、数据权限管理的菜单、按钮和后端 API 权限码。
- 提供可被业务服务复用的数据权限模型，为后续 `common-security` 下沉执行打基础。

**Non-Goals:**

- 不在第一版完成所有微服务的数据权限接入。
- 不做用户级特殊数据权限配置页面，仅预留模型扩展空间。
- 不实现复杂 ABAC 表达式引擎。
- 不引入外部策略引擎，如 OPA、Casbin。
- 不重构现有用户、角色、菜单、按钮权限主链路。

## Decisions

### D1: 组织对象与组织关系分离

`sys_org_unit` 表达组织对象本身，`sys_org_dimension` 表达组织树视角，`sys_org_relation` 表达某个维度下的父子关系和路径。

理由：

- 单个 `parent_id` 无法表达同一组织对象在行政、法人、核算、业务维度下的不同父级。
- 独立关系表便于后续支持组织树移动、维度切换和闭包表优化。

备选：

- 只在 `sys_org_unit` 增加 `org_type` 和 `parent_id`。该方案简单，但会把组织对象类型和组织树类型混在一起，后续扩展成本高，放弃。

### D2: 用户组织归属按维度存储

`sys_user_org_unit` 增加 `dimension_id`、`is_primary`、`position_name`、`is_leader`、`status` 等字段。第一版默认使用 `ADMIN` 行政组织作为用户主归属。

理由：

- 用户可能同时存在行政归属、项目归属和业务归属。
- 主归属用于默认查询、通讯录展示和后续审批发起。

### D3: 数据权限先做角色级策略

数据策略表保留 `subject_type` 字段，但第一版前端只开放 `ROLE` 策略配置。

理由：

- 角色级策略能覆盖大多数企业权限治理场景。
- 用户级特例容易破坏可解释性，第一版只预留，避免产品复杂度过早膨胀。

### D4: 多维组织交叉默认收窄

同一维度内多个组织默认 OR；不同维度之间默认 AND；多角色之间默认 OR；显式 DENY 优先。

理由：

- 同维度多个授权组织表示多个可见范围。
- 跨维度交叉涉及财务、合同、审批等高敏数据，默认 AND 更安全。

### D5: 权限码继续使用 API 权限标准

组织管理、数据权限管理的页面 GET 权限挂在页面菜单节点；新增、修改、删除、保存等写操作使用 button 节点绑定对应 API 权限码。

理由：

- 与现有用户管理、角色管理、菜单管理授权逻辑保持一致。
- 搜索、刷新、分页、详情复用页面 GET 权限，不额外创建搜索按钮权限。

### D6: 数据权限执行先提供策略计算服务

第一版后端实现策略 CRUD、角色策略查询和标准数据范围 DTO。业务服务可通过统一服务获取策略并自行转换为查询条件。

理由：

- 当前平台尚未抽出完整 `common-security` 模块，直接强制所有服务接入会扩大风险。
- 先统一数据模型和策略解析，再逐步下沉到公共组件。

## Risks / Trade-offs

| Risk | Mitigation |
|------|------------|
| 组织模型迁移影响现有 `sys_org_unit.parent_id/tree_path` | 保留旧字段兼容，新增维度关系表并用迁移初始化 `ADMIN` 维度树 |
| 多维数据权限前端配置复杂 | 第一版只开放角色级策略和常用范围类型 |
| 数据权限模型已建但业务服务尚未全部接入 | 提供标准策略查询 API 和 DTO，分阶段接入各业务资源 |
| 多组织树查询性能下降 | 先用 `tree_path` 支撑范围展开，后续高频场景再引入闭包表 |
| 权限缓存尚未完全实现 | 本阶段保留文档与模型，代码先兼容现有权限头和本地服务校验 |

## Migration Plan

1. 新增 Flyway 迁移，创建 `sys_org_dimension`、`sys_org_relation`、`sys_data_policy`、`sys_data_policy_dimension`。
2. 扩展 `sys_org_unit` 和 `sys_user_org_unit` 字段，保留历史字段兼容。
3. 初始化 `ADMIN`、`LEGAL`、`ACCOUNTING`、`BUSINESS` 组织维度。
4. 将现有 `sys_org_unit.parent_id/tree_path` 数据初始化为 `ADMIN` 维度关系。
5. 新增组织管理和数据权限管理菜单、按钮和 API 权限种子数据。
6. 实现后端 API 和前端页面。

回滚策略：新增表和字段均为增量变更，不删除现有字段。若需要回退功能，可禁用新增菜单并保留数据表，不影响现有用户、角色、菜单能力。

## Open Questions

所有本阶段关键范围已确认：

1. 独立 change：`org-data-permission-foundation`。
2. 第一版数据权限只开放角色级配置，用户级策略预留。
3. 前端入口放在“系统管理”下，新增“组织管理”和“数据权限”两个菜单。
