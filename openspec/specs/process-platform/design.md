## MVP 收口决策（v4）

### 流程包所有权

- `service-workflow-engine` 拥有流程包、版本、发布状态、实例绑定和运行时解释权。
- `service-lowcode` 只拥有可复用表单定义，不拥有流程包生命周期。
- 草稿可以引用 lowcode 表单 ID；发布时 workflow-engine 通过内部只读接口获取已发布表单。

### 发布快照

- 发布事务同时校验流程图、参与者、条件、JSON Schema 和 UI Schema。
- 发布版本固化 `processJson`、`formSchema`、`formUiSchema` 和表单定义版本。
- 运行中实例始终绑定启动时的流程包 ID 与版本，不追随后续版本变化。

### 候选任务

- 普通审批节点只创建一条任务，并在 `wf_task_candidate` 固化节点进入时的候选人集合。
- 第一个合法候选人通过条件更新原子认领任务；非候选人和并发失败者不能改变任务或 Workflow 状态。
- 会签节点按参与者创建独立任务，由 Workflow 维护 ALL/ANY 计票状态。
- 退回节点会产生新的访问序号和新的参与者快照。

### MVP 节点范围

- 可发布节点仅为 `START`、`APPROVAL`、`COUNTERSIGN`、`END`。
- 条件是节点连线属性，通过受限表达式 Activity 执行，不作为独立可发布节点。
- `NOTIFY`、`TIMER`、`SUB_PROCESS`、`SERVICE_TASK`、`SAGA`、`EVENT` 等保留在后续阶段，设计器可以展示但必须阻止发布。

## 定位

流程平台不是"审批功能"，而是 **TrioBase 流程基座的核心载体，是企业级业务系统的运行时底座**。

核心原则：**先建底座，再盖业务系统。**

```

                    │     三套上层业务系统（后续建设）       │

                    │  │ SCM  │  │ OMS  │  │ WMS  │      │
                    │  └──┬───┘  └──┬───┘  └──┬───┘      │
                    └─────┼──────────┼──────────┼─────────┘
                          │          │          │
                    ┌─────▼──────────▼──────────▼─────────┐
                    │    流程平台运行时底座（本次建设）       │
                    │                                     │

                    │  │  一次定义，三流通用            │    │
                    │  │  ┌─────┐ ┌─────┐ ┌────────┐ │    │
                    │  │  │审批流│ │业务流│ │集成流  │ │    │
```

### 流程平台的三流归一

| 流类型 | 典型场景 | 节点特征 | 谁消费 |
|--------|----------|---------|--------|
| **审批流** | 报销审批、采购审批、合同审批 | 以人审为核心 (APPOVAL/COUNTERSIGN/加签/转办) | 企业员工 |
| **业务流** | 订单状态机、入库/出库流程、履约分发 | 以系统执行为核心 (SERVICE_TASK/SAGA/CONDITION/EVENT) | 业务系统 (SCM/OMS/WMS) |
| **集成流** | 订单→WMS 分发、支付回调处理、ERP 同步 | 以外部系统协作为核心 (ASYNC_SERVICE/EVENT/TRANSFORM) | 外部系统/中间件 |

这三流共享同一套**流程包定义体系**，只是节点类型和参与者模式不同。设计时不区分，运行时按节点类型自动路由。

## 底座建设范围（本次要做的事）

本次只建底座能力，不包含任何业务系统功能：

```
✅ 做                        ❌ 不做
──────────────────────────────────────────
流程包定义与管理              SCM 特有的采购管理逻辑
表单设计器 + 表单运行时        OMS 的订单计算/计费
流程引擎 (三流统一)            WMS 的库位管理/波次策略
权限体系 (含行级数据权限)      业务专用报表
参与者解析                    业务定制化 UI
待办任务中心                   
扩展服务 (打印/附件/通知)      
数据沉淀层                    
审计日志                    
```

**底座建设完成后，上层业务系统通过以下方式接入：**
1. 定义流程包（设计器或代码 JSON）
2. 注册业务 Service（节点解析器调用的后端能力）
3. 配置权限策略
4. 消费数据沉淀（宽表/事件）

## 核心概念

### 流程包（Process Package）

流程包是设计层的唯一产物，也是运行时的唯一输入。**审批流、业务流、集成流共用同一套定义格式**，区别仅在于节点类型组合不同。

```json
{
  "processPackage": {
    "version": "1.0.0",
    "processKey": "expense_report",
    "name": "费用报销",
    "category": "approval",    // 分类标记：approval/business/integration
    
    "form": {
      "schema": { /* JSON Schema: 字段定义、校验规则 */ },
      "uiSchema": { /* 布局配置: 栅格、分组、子表单 */ }
    },
    
    "flow": {
      "nodes": [
        { "id": "start", "type": "START", "next": "dept_approve" },
        { 
          "id": "dept_approve", 
          "type": "APPROVAL",
          "assignment": { "type": "ROLE", "roleCode": "DEPT_HEAD" },
          "next": [
            { "condition": "amount > 5000", "target": "finance_approve" },
            { "condition": "true", "target": "end" }
          ]
        },
        { "id": "finance_approve", "type": "COUNTERSIGN",
          "strategy": "ALL"
        },
        { "id": "end", "type": "END" }
      ]
    },
    
    "permissions": {
      "start": ["ROLE:USER"],
      "view": ["ROLE:USER", "ROLE:DEPT_HEAD"],
      "data": [   // 行级数据权限
        { "resource": "instances", "rule": "department = currentUser.department" }
      ],
      "admin": ["ROLE:ADMIN"]
    },
    
    "extends": {
      "printTemplates": [{ "id": "default", "template": "..." }],
      "attachments": { "maxCount": 10, "maxSize": "20MB" },
      "notifications": [
        { "on": "node_enter", "to": "assignee", "channel": "internal" }
      ]
    }
  }
}
```

### 设计态与运行态分离

```
设计态                                           运行态
─────────                                       ─────────
流程包 JSON  (版本化, 不可变)                     流程实例 (ProcessInstance)
  │                                                 │
  │  ┌─ 表单定义 ────────────────────────────┐      │  ┌─ 表单数据快照 ─────────┐
  │  │ 字段定义、校验、布局、默认值           │      │  │ 用户填写的具体值        │
  │  └──────────────────────────────────────┘      │  └────────────────────────┘
  │  ┌─ 流程定义 ────────────────────────────┐      │  ┌─ 流转状态 ─────────────┐
  │  │ 节点、连线、条件、参与规则、超时       │      │  │ 当前节点、历史记录      │
  │  └──────────────────────────────────────┘      │  └────────────────────────┘
  │  ┌─ 扩展配置 ────────────────────────────┐      │  ┌─ 运行时数据 ───────────┐
  │  │ 打印模板、通知规则、附件限制           │      │  │ 附件列表、审批记录、    │
  │  └──────────────────────────────────────┘      │  │ 操作日志、通知记录      │
  │                                                 │  └────────────────────────┘
  │  发布即冻结                                    │  每个实例独立
  │  新版本通过升级产生                             │  实例存活于整个流转周期
```

### 3. 流程运行时模块

这是底座的核心——也是三流归一的地方。审批流、业务流、集成流的区别只体现在节点类型上，引擎本身统一处理所有流程实例的生命周期。

```
┌──────────────────────────────────────────────────────────────┐
│  流程运行时                                                    │
├──────────────────────────────────────────────────────────────┤
│                                                               │
│  1. 流程实例引擎                                               │
│     根据流程包定义创建流程实例                                  │
│     管理实例生命周期 (创建→活跃→挂起→完成→终止)               │
│     支持 SAGA 事务补偿（失败节点 → 反向触发补偿节点）          │
│                                                               │
│  2. 节点解析器                                          ← 核心 │
│                                                               │
│     ┌──────────────┬────────┬──────────────────────────┐      │
│     │ 节点类型      │ 参与者  │ 行为                    │      │
│     ├──────────────┼────────┼──────────────────────────┤      │
│     │ START        │ 系统    │ 初始化实例，进入下一节点 │      │
│     │ APPROVAL     │ 人      │ 创建待办任务，等待 Signal│      │
│     │ COUNTERSIGN  │ 多人    │ 会签/或签/按比例通过     │      │
│     │ NOTIFY       │ 人/系统 │ 发送通知，自动继续       │      │
│     │ CONDITION    │ 系统    │ 计算表达式，决定流转方向 │      │
│     │ SERVICE_TASK │ 系统    │ 调用后端API/内部服务     │      │
│     │ SAGA         │ 系统    │ 分布式事务：正向+反向补偿│      │
│     │ EVENT        │ 系统    │ 等待外部事件/异步回调    │      │
│     │ SUB_PROCESS  │ 系统    │ 启动子流程，等待完成     │      │
│     │ TIMER        │ 系统    │ 等待超时或定时到达       │      │
│     │ GATEWAY      │ 系统    │ 并行/排他/包容网关       │      │
│     │ SCRIPT       │ 系统    │ 执行脚本逻辑             │      │
│     │ END          │ 系统    │ 结束实例                 │      │

│                                                               │
│  3. 参与者解析器                                               │
│     ┌────────────┬───────────────────────────┐               │
│     │ 模式        │ 解析方式                  │               │
│     ├────────────┼───────────────────────────┤               │
│     │ ROLE       │ service-auth 角色→用户列表│               │
│     │ DEPT       │ service-org 部门→用户列表 │               │
│     │ USER       │ 指定用户                  │               │
│     │ DYNAMIC    │ 从表单数据提取             │               │
│     │ MANAGER    │ 发起人上级（查组织树）     │               │
│     │ SYSTEM     │ 无参与者，自动执行         │               │
│     └────────────┴───────────────────────────┘               │
│                                                               │
│  4. 待办任务管理器                                             │
│     适用于有人参与的节点 (APPROVAL/COUNTERSIGN/NOTIFY)        │
│     创建/完成/转办/加签/撤回                                  │
│     待办列表（按用户/角色/部门/仓库等维度）                    │
│     已办列表 + 审批历史穿透                                    │
│                                                               │
│  5. 超时/升级管理器                                            │
│     ⏰ 定时器节点：等待指定时间后自动触发                       │
│     ⬆ 超时升级：节点超时 → 自动转审/通知                      │
│     📋 流程时限：整体流程超时 → 自动升级                       │
│                                                               │
│  6. 驳回/回退                                                 │
│     驳回: 退回指定节点（可带表单数据修改）                     │
│     回退: 退回上一步                                          │
│     撤回: 发起人撤回未处理的流程                               │
│                                                               │
│  7. SAGA 补偿引擎                                              │
│     为 SAGA 节点类型服务                                      │
│     正向：按顺序执行 N 个服务任务                              │
│     反向：任一失败 → 反向触发对应对的补偿节点                   │
│     补偿节点在流程包中显式定义                                 │
│                                                               │
│  8. 外部事件监听器                                             │
│     为 EVENT 节点类型服务                                     │
│     订阅 Kafka/Pulsar 等事件总线                              │
│     事件到达 → 匹配等待中的流程实例 → 恢复流转                 │
│                                                               │
└──────────────────────────────────────────────────────────────┘
```

### 5. 数据沉淀模块

数据沉淀层是底座提供给上层业务系统的"数据红利"——流程跑出来的数据不只是在流程引擎里，而是结构化输出到分析层。

```
┌──────────────────────────────────────────────────────────────┐
│  数据沉淀层                                                    │
├──────────────────────────────────────────────────────────────┤
│                                                               │
│  1. 流程实例宽表                                               │
│     已完结流程 → 表单字段解构为列                              │
│     每行 = 一个流程实例 + 完整表单字段展开                      │
│     原生 SQL 可查询（不需要 JSON 解析）                         │
│                                                               │
│     例如：费用报表                                         │
│     ┌──────────┬──────┬────────┬────────┬──────────┐       │
│     │ instance_│amount│dept    │approver│duration  │       │
│     │ id       │      │        │        │          │       │
│     ├──────────┼──────┼────────┼────────┼──────────┤       │
│     │ EXP-001  │ 3800 │ 技术部  │ 张三   │ 2h       │       │
│     │ EXP-002  │12000│ 市场部  │ 李四   │ 3d       │       │
│     └──────────┴──────┴────────┴────────┴──────────┘       │
│                                                               │
│  2. 流程效率分析                                               │
│     各节点平均耗时、驳回率、超时率                              │
│     按组织维度统计（部门/角色/人员）                            │
│     按流程维度统计（哪个流程效率低？）                          │
│                                                               │
│     未来价值（SCM/OMS/WMS 接入后）：                          │
│     - SCM: 供应商履约周期分析 / 采购效率优化                    │
│     - OMS: 订单处理时长分析 / 异常订单根因                     │
│     - WMS: 入库出库效率分析 / 波次策略效果                     │
│                                                               │
│  3. 数据输出管道                                               │
│     流程完结后 → 结构化数据推送到 Kafka/Pulsar                │
│     下游订阅方:                                               │
│     ├─ BI 系统 (ClickHouse 等)                                │
│     ├─ AI 训练 (特征工程)                                     │
│     └─ 业务系统 (SCM/OMS/WMS 自身分析)                       │
│                                                               │
│  4. 后续扩展方向 (底座建设完成后)                               │
│     流程数据 → 特征工程 → 预测 (预计耗时/审批概率/异常检测)   │
│     AI 分析流程效率数据 → 推荐优化流程定义                     │
│                                                               │
└──────────────────────────────────────────────────────────────┘
```

## 决策记录

### D1: 流程运行时底座 — 保留 Temporal，不引入 Flowable

| 方案 | 结论 |
|------|------|
| 自建简化版状态机 + DB 事务 | ❌ 放弃（SCM/OMS/WMS 需要的 SAGA/进程恢复/事件等待全部手写，成本过高） |
| **Temporal** | ✅ **确定使用**（docker-compose 已有容器，pom.xml 已有 SDK，三流归一的唯一路径） |
| Flowable/Activiti | ❌ 放弃（审批流开箱虽好，但业务流/集成流/跨语言协调能力不足，引入双引擎成本更高） |

> 注意：Temporal 不提供审批节点开箱能力（待办/会签/驳回/加签）。这些需要作为 Activity 实现在 Temporal Workflow 之上。这是一次性建设投入，换来的是一条路走到底。

### D2: 流程包定义格式 — 自定制 JSON Schema

- ✅ **确定：自定制 JSON Schema**（灵活、设计器原生消费、AI 易生成、版本化方便）

### D3: 参与者解析 — 节点创建时固化

- ✅ **确定：节点创建时固化参与者快照**
- 理由：组织调整不应影响已流转到节点的审批人列表
- 后续按节点级别配置"实时查询"选项

### D4: 会签策略 — 平行会签默认

- ✅ **确定：平行会签**（N 人同时审批，计票通过）
- 后续可按节点配置串行会签

### D5: 流程版本升级 — 已有实例走旧版

- ✅ **确定：新版本发布后，已有实例继续走旧版定义**
- 提供可选的管理操作"迁移到新版"

### D6: 数据沉淀触发 — 双轨策略

- ✅ **确定：主动触发 + 定时 ETL 双轨**
- 流程完结时主动推送事件（实时供下游消费）
- 定时任务解构宽表（供 BI 分析）
- 初期先做主动触发

### D7: 流程设计器 — X6

- ✅ **确定：AntV X6** 作为流程画布引擎
- 项目已有 Ant Design Vue 生态兼容
- 不绑定业务语义（审批/业务/集成流程图共用同一套画布组件）

### D8: 现有代码基复用策略

| 现有模块 | 复用方式 | 改造量 |
|----------|---------|--------|
| `service-auth` | 认证/权限/参与者解析直接调用 | 小（注册流程相关权限码+菜单） |
| `service-org` | 组织树查询/部门参与解析直接调用 | 小（接口已够） |
| `service-ops` | 消息→通知引擎，文件→附件服务 | 小（业务流程中调 API） |
| `service-lowcode` | 表单定义、表单实例、低代码应用元数据 | 中（强化租户/版本/发布快照/泛化运行时） |
| `service-workflow-engine` | 流程包生命周期、流程运行时、业务闭环执行 | 大（核心流程编排与发布快照） |
| `platform-gateway` | Token 校验/用户上下文透传 | 无（已完备） |

## 底座建设的阶段划分

> 基于决策 D1（直接上 Temporal），三阶段重新定义如下：

### 第一阶段：最小闭环 — 跑通第一个审批流

```
目标: 从"后端创建流程包 → 用户提交 → 审批流转 → 完成"全链路跑通
时长估算: 2-3 周

后端（2 周）:
  ├─ service-workflow-engine 流程包表设计 + CRUD API（含版本/发布/下架）
  ├─ service-lowcode 表单定义与已发布表单快照只读接口
  ├─ service-workflow-engine 骨架搭建
  │   ├─ 启用 Temporal Worker（temporal.enabled=true）
  │   ├─ 第一个 Workflow: ProcessWorkflow
  │   │   ├─ 接收流程包定义 → 逐节点流转
  │   │   ├─ START → APPROVAL（Signal 等待审批）→ CONDITION → END
  │   │   └─ Activity: CreateTask, ResolveParticipants, EvaluateCondition
  │   ├─ 流程实例表 (wf_process_instance)
  │   └─ 待办任务表 (wf_task)
  ├─ 参与者解析: ROLE 模式（调 service-auth）
  ├─ DB 迁移脚本（V1__workflow_engine_schema.sql）
  └─ 测试用流程包种子数据（费用报销）

前端（1 周）:
  ├─ 流程包管理页面（列表/新建/编辑/发布）
  ├─ 流程实例管理页面（运行中/已完成）
  ├─ 待办任务页面（我的待办/我已办理）
  └─ 表单动态渲染（基于流程包 schemaJson）

产出: 一个完整的审批流（费用报销）可以跑通
```

### 第二阶段：流程引擎完备

```
目标: 补齐审批流全部能力 + 扩展服务对接
时长估算: 3-4 周

后端（3 周）:
  ├─ 节点类型补齐
  │   ├─ COUNTERSIGN（平行会签/或签/按比例）
  │   ├─ NOTIFY（自动发送通知，调 service-ops）
  │   ├─ TIMER（超时升级）
  │   ├─ SUB_PROCESS（子流程）
  │   └─ SERVICE_TASK（调后端 API）
  ├─ 参与者解析补齐: DEPT / USER / DYNAMIC / MANAGER
  ├─ 驳回 / 回退 / 转办 / 加签
  ├─ 参与者快照固化（节点创建时写入任务表）
  ├─ 审批历史表 + 操作审计日志
  └─ 对接扩展服务
      ├─ 通知引擎 → service-ops message API
      └─ 附件服务 → service-ops file API

前端（1 周）:
  ├─ 流程设计器（X6 画布 + 节点配置面板）
  │   ├─ 左侧节点面板（拖拽到画布）
  │   ├─ 右侧属性面板（节点配置/参与人/条件表达式）
  │   ├─ 连线（条件标签）
  │   └─ 导出/导入流程包 JSON
  ├─ 会签组件（显示计票进度）
  ├─ 转办/加签/驳回的操作 UI
  └─ 审批历史时间线

产出: 审批流功能完备，可支撑企业级审批业务
```

### 第三阶段：业务流 + 集成流 + 数据

```
目标: 三流归一 + 数据沉淀
时长估算: 4-6 周

后端（4 周）:
  ├─ SAGA 节点（分布式事务补偿）
  │   ├─ 流程包定义中显式声明补偿节点
  │   ├─ Temporal Workflow 中 Try/Catch 触发补偿
  │   └─ 补偿执行日志
  ├─ EVENT 节点（外部事件等待）

```
